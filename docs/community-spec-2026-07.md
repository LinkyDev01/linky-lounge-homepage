# 레이지데이 북클럽 — 회원 커뮤니티화 기능 명세 (2026-07 초안)

> 4개 관점(멤버 UX·운영 어드민·기술 아키텍처·법무 리스크) 병렬 분석 + 교차 비평으로 작성.
> **이 문서는 명세 초안이며 실행 지시가 아니다** — 각 항목은 운영자 승인 후 프리뷰 퍼스트 절차로 착수한다.
> 규칙 원본은 `CLAUDE.md`·`docs/DECISIONS.md`. 디자인 브리프는 `docs/design-brief-2026-07.md`.

## 0. 설계 원칙 (모든 기능의 잣대)

1. **웹의 역할은 셋뿐** — 문(초대 입회라는 의식), 시계(요일 선택·리마인더), 서랍(사후 기록·아카이브). 이 셋을 벗어나는 기능은 대부분 오프라인 침식이다.
2. **로그인 벽 자체가 기능이다** — 공개 '회원가입' 버튼이 없는 사이트. 인터뷰 합격자에게만 문이 열리는 구조가 '선별의 서사'의 기술적 완성형.
3. **모더레이션할 대상을 만들지 않는다** — 멤버의 웹 입력은 구조화된 클릭(요일 선택, 재등록)으로 한정. 자유 텍스트·대화는 라운지와 손글씨에 남긴다. 1인 운영의 모더레이션 비용을 0으로 수렴시키는 원칙.
4. **시점 설계가 콘텐츠 규칙의 전부** — 오프라인보다 먼저 소비되는 콘텐츠를 두지 않는다 (발제문은 회차 종료 후, 철학 원고는 입회 후 등).
5. **자동화의 경계** — 시스템은 집계·감지·발송 준비까지, 취소·환불·합격 같은 결정과 대외 발화는 운영자 원탭 승인으로.

## 1. 릴리스 절단선 (권고)

6개 관점의 MVP를 합치면 1인 운영+3기 병행으로 8주 안에 불가능하다. 3기 도중 도입은 진행 중 회차와 충돌하므로 **적용 시점은 4기부터**를 권고. 절단선:

| 릴리스 | 내용 | 시점 |
|---|---|---|
| **0차 (즉시, 커뮤니티화와 무관)** | 개인정보처리방침 제정·게재 + 신청 폼 동의 분리(필수/마케팅 선택) — **현행 부채**: 현 약관 제15조가 존재하지 않는 처리방침을 참조 중이고, 통합 동의가 마케팅을 필수로 강제 중 | 지금 |
| **1차 (4기 개막 전)** | DB·인증 기반 + 초대 입회 + 회차별 요일 선택 + 멤버 홈 + 재입회 우선권 | ~9월 |
| **2차 (4기 중)** | 토스페이먼츠 입회 결제 일체화 + D-1 리마인더 + 레이지노트 사후 아카이브 + 나의 서가 | 4기 중 |
| **3차 (5기~)** | 기수 전환 콘솔(어드민) + 기수의 기록 + 멤버 카드 + 휴면 지위 + 알럼나이 초대석 | 장기 |

## 2. 기능 명세

### 2-1. 입구 (문)

| 기능 | 요지 | 릴리스 |
|---|---|---|
| **초대 입회** | 인터뷰 합격자에게만 만료 기한 있는 1회용 입회 링크 발급(알림톡). 사이트 어디에도 공개 가입 버튼 없음. 링크 진입 → 약관·개인정보 동의(만 14세 확인 포함) → 계정 연결 → 멤버 확정. **만료 링크 재발급·인증 실패의 예외 흐름을 반드시 함께 설계** (20명 규모에서 온보딩 사고 1건 = 멤버의 5%, 이 브랜드에선 CS 티켓이 아니라 서사 붕괴) | 1차 |
| **인증 (추천: 카카오 로그인 단일)** | Supabase Auth의 카카오 프로바이더 하나만. 자체 비밀번호 없음. 초대 링크의 클레임 토큰으로 계정-합격자 자동 연결. 근거: ①자체 OTP 구현(발급·검증·rate limit)을 피함 ②알림톡과 '카카오로 들어와 카카오로 알림받는' 닫힌 루프 ③전화번호 scope 심사 리드타임 회피. 대안: 전화번호 SMS OTP(플랫폼 냄새는 더 적으나 자체 인증 구현 부담) — **운영자 선택 필요** | 1차 |
| **입회 결제 일체화** | 입회 링크 → 동의 → 토스페이먼츠 결제 → 권한 부여를 한 흐름으로. 결제 완료가 '멤버 확정'의 단일 트리거. 결제 화면 환불 안내는 약관과 표현 일치 원칙(`docs/tosspayments-refund-notice.md`) 준수. 1차에서는 계좌이체 병행+수동 확인 버튼으로 시작 가능 | 2차 |
| **재입회 우선권** | 기수 수료 멤버는 인터뷰 없이, 공개 모집 전 우선 신청 기간에 '4기 계속하기' 원탭. 할인·리워드가 아니라 **'먼저 열리는 문'**의 형태. ⚠ 정원 배분 규칙(우선권 몇 석 / 신규 몇 석) 운영자 결정 필요 — 우선권이 정원을 다 채우면 신규 선별이 공회전 | 1차 |

### 2-2. 시계 (일정)

| 기능 | 요지 | 릴리스 |
|---|---|---|
| **회차별 요일 선택 (RSVP)** | 정규 4회차 각각 수/목/일 선택·변경(회차 D-2까지). 요일별 잔여 인원 숫자는 비노출(잔여석 비표기 확정 결정과 동일 원칙)하되, **'이 요일은 인원이 적습니다' 수준의 중간 신호는 필요** — 기습 취소가 숫자 노출보다 나쁨. 일요일(마지막 요일)은 인원 무관 진행이라는 약관 예외를 로직·문구에 반영. ⚠ 5회차(자유모임) RSVP는 별도 정의 필요 | 1차 |
| **요일 현황판 (어드민)** | 회차×요일 집계 매트릭스. 2명 이하 요일 자동 플래그 → 운영자가 '취소' 버튼 → 대상자 알림톡 일괄 발송+재선택 유도. 자동 집행은 금지(운영자 승인 필수) | 1차 |
| **D-1 리마인더** | 내가 고른 요일 기준 전날, 시간·장소·이번 책을 담은 알림톡 **회차당 정확히 1통**. 푸시·뉴스레터 없음. 좋은 호텔의 컨시어지 문자 문법 | 2차 |
| **출석 원탭 체크인 (어드민)** | 모임 당일 운영자가 RSVP 명단에서 탭 체크. QR·셀프 체크인 없음. ⚠ 반복 노쇼 정책(요일 집계 왜곡 방지) 운영자 결정 필요 | 2차 |
| **서신형 공지** | 운영자만 발행, 멤버는 열람만. 댓글·반응·조회수 없음. '알림톡으로도 보내기' 체크 하나 | 1차 |

### 2-3. 서랍 (기록·아카이브)

| 기능 | 요지 | 릴리스 |
|---|---|---|
| **멤버 홈 (한 장)** | 로그인 후 도착하는 단일 페이지: 다음 모임(내 요일·D-day)·이번 책·장소만. 대시보드·위젯·피드 없음. **첫 로그인 화면의 밀도를 별도 설계할 것** — 선별 서사를 세워놓고 문 뒤가 비어 있으면 낙차가 큼 | 1차 |
| **레이지노트 사후 아카이브** | 발제문을 해당 회차의 **모든 요일 모임 종료 후에만** 멤버 전용 열람 공개(사전 공개 금지 — 현장의 긴장 보존). 다운로드 없이 열람 위주. ⚠ 도서 인용 저작권(제28조 정당 인용 범위) 점검 기준 필요 | 2차 |
| **나의 서가** | 참여 기수들의 책이 큐레이터 노트와 함께 쌓이는 개인 페이지. 통계·달성률 없이 책등이 늘어나는 조용한 시각화. 기수제의 '8주 리셋' 약점을 잇는 연속성 자산 | 2차 |
| **기수의 기록** | 기수 종료 시 생성되는 아카이브 한 장 — 네 권, 다섯 날짜, (동의한) 멤버 이름. 수료증·배지가 아닌 기록물. 페이지 끝에 재입회 안내가 자연스러운 다음 문장이 되는 구조 | 3차 |
| **멤버 카드** | 이름+입회 기수('3기부터') 각인의 디지털 카드 한 장. 공유 가능하되 유도하지 않음 | 3차 |
| **휴면 지위 ('쉬는 중')** | 다음 기수 미등록 시에도 계정·서가 열람권 유지, 일정 기수 내 복귀 시 인터뷰 면제 유지. 이탈을 '탈퇴'가 아니라 '쉬는 기수'로 — 브랜드명('게으른 날')과 공명 | 3차 |
| **알럼나이 초대석** | 5회차 자유모임에 이전 기수 멤버 소수 초대(좌석 수는 운영자가 기수마다 설정, 0 가능) | 3차 |

### 2-4. 기술 기반 (추천 스택)

- **DB: Supabase Postgres** (추천 — Auth·Storage·RLS 일체, 이 규모에서 무료 티어로 수년). 핵심 엔티티 6: members / seasons / sessions / attendances / payments / applications.
- **GAS 스트랭글러 공존**: 즉시 폐기 금지. 1단계 신청 제출을 Next.js Route Handler가 받아 DB 기록 후 GAS로 포워딩(알림톡·시트는 GAS 유지) → 점진 이관. 시트는 **읽기 사본**으로 강등(양방향 동기화 금지 — 드리프트 사고 경로).
- **세션**: @supabase/ssr 쿠키 세션, 기존 middleware.ts에 체이닝. 어드민은 role 컬럼+RLS로 이관(현 쿠키 시크릿은 과도기 병행).
- **월 비용**: Vercel Pro $20(상업 운영 요건) + Supabase 0원 + 알림톡 건당 과금 수준.
- **비-UI 변경(DB·인증·결제)의 승인 게이트 절차를 CLAUDE.md에 신설 필요** — 프리뷰 퍼스트는 정적 UI 전제라 시드 데이터·테스트 결제·프리뷰용 인증 우회 방식을 먼저 정의해야 함.

### 2-5. 법무 (0차 = 현행 부채 포함)

| 항목 | 요지 | 릴리스 |
|---|---|---|
| 개인정보처리방침 제정 | 법 제30조 기재사항 + 처리위탁(Sheets/GAS·Vercel·Solapi·토스) 명시, 푸터 게재. **현 약관 제15조가 참조하는 문서가 실제로 없음** | **0차 (즉시)** |
| 동의 분리 | 현행 통합 체크박스를 ①수집·이용(필수) ②마케팅(선택) ③사진·콘텐츠 활용(선택)으로 분리, 항목별 consentAt | **0차 (즉시)** |
| 만 14세 확인 | 가입 게이트 + 약관 조항 신설 | 1차 |
| 약관 커뮤니티 장 신설 | 현행 17조(참가·환불)는 불변, 계정·게시물·금지행위·임시조치 장 추가. 개정은 제3조 절차(불리 개정 30일 전 게시) | 1차 |
| 기존 신청자 데이터 전용(轉用) 재동의 | 1·2기 신청 데이터는 '접수' 목적 수집분 — members로 승계하려면 **초대 링크 수락 시점에 재동의** 흐름 필수 | 1차 |
| 탈락자 데이터 파기 | 선별 구조의 법적 그림자 — 탈락자 개인정보의 파기 시점·절차 명문화. 탈락 사유 DB화 금지 | 1차 |
| 안전성 확보조치 | DB 생성과 **동시** 발생 의무(RLS·암호화·접속기록) — 후속 과제 아님 | 1차 |
| 통신판매업 신고 | 토스페이먼츠 PG 심사의 선행 조건 여부 확인 + 푸터 법정 표시 | 2차 선행 |
| 광고성 정보 요건 | 4기 모집 알림톡 = 광고성 → (광고) 표기·무료 수신거부·야간 금지. **3기 종료 전 정비** | 1차 |
| 초상권·촬영 동의 | 기수 시작 시 촬영·게재·외부 활용 각각 선택 동의, 철회 절차, 미동의자 프레임 제외 운영 | 2차 |

### 2-6. 넣지 않는 것 (Anti-features, 전 관점 합의)

공개 피드·타임라인(좋아요·댓글) / 상시 단체 채팅·DM / 게이미피케이션(배지·챌린지·리더보드·포인트·출석 스트릭) / 독서 진도·완독 인증 / 공개 멤버 디렉토리·프로필 / 발제문 사전 공개 / 잔여석 카운터·마감 임박 연출 / 추천인 리워드 / 별점·온라인 후기 폼 / 웹 푸시·인앱 알림 / 자체 비밀번호 인증 / 별도 백엔드 서버·마이크로서비스 / 네이티브 앱.

공통 논리: 온라인이 오프라인 대화를 선소비하는 것, 성취 압박('게으른 날'과 충돌), 숫자 과시(선별 서사와 충돌), 1인 운영이 감당 못 하는 모더레이션·인프라.

## 3. 운영자 결정 필요 목록 (명세 확정 전)

1. **적용 시점**: 4기부터 권고 — 동의 여부.
2. **인증 방식**: 카카오 로그인 단일(추천) vs 전화번호 OTP.
3. **철학 원고('결'·'불균형의 균형')의 지면**: ⓐ공개 스테이트먼트 페이지(자기 선별 필터) ⓑ멤버 전용(배타성 자산) ⓒ인터뷰 예약 페이지 — 상호 배타적, 하나만 가능.
4. **환불 기준일의 개인별 상대성**: 요일 선택제에서 '회차 시작일'이 기수 공통(가장 이른 날 — 현 약관 제2조⑦ 해석)인지 본인 선택 요일 기준인지 — 환불 자동 계산의 전제. 현 약관은 공통 기준일로 읽히나 확인 필요.
5. **정원 배분**: 재입회 우선 좌석 수 vs 신규 모집 좌석 수.
6. **막힌 순간의 문의 채널**: DM 배제 상태에서 OTP 미도달·결제 실패 시 출구 하나(예: 전용 이메일 or 카카오 채널 1:1) 지정.
7. **'모임 후 한 줄'(멤버 자유 텍스트) 허용 여부**: 원칙 3(구조화 입력만)과 후기 자산 파이프라인 사이의 트레이드오프.
